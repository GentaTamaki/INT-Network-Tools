# 7. タイミングモデルと Receiver の動作

Receiver は以下のロジックで受信パケットを処理します。

1. SEQ によるパケット順序管理
2. パケットロス検出
3. ACTIVE フィールドによるソース解決
4. Resolved Time の生成
5. Output Policy に基づく出力生成

Receiver 実装は、必要に応じて以下のようなタイミングモデルを実装することがあります。

- Holdover
- Smooth Reacquire
- Reference Discipline

## 7.1 Sender 送信モデル

INTTC Sender は、プロジェクトのフレームレートとは独立した固定周期でパケットを送信します。

デフォルトの Sender モード：

- パケットレート: 60 Hz
- 想定送信間隔: 約 16.67 ms

各パケットは、送信時点における **Sender の論理タイミング状態のスナップショット** を表します。

パケットには以下の情報が含まれます。

- primary timecode
- alternate timecode
- sender state
- active source
- frame rate
- drop-frame flag

INTTC パケットは **1フレームごとのメッセージとして定義されるものではありません**。また、将来フレームをスケジューリングする指示として解釈してはいけません。

Receiver はパケット到着時刻ではなく、**パケット内容をタイミング状態の基準**として扱う必要があります。

Sender は、重要な状態変化が発生した際に **即時パケット** を送信することがあります。代表的なトリガーは以下です。

- 再生 / 停止の切り替え
- ソース選択の変更
- タイムコードの不連続
- フレームレート変更

即時パケットにより、Receiver は次の周期送信を待たずに状態変化へ素早く反応できます。

## 7.2 Receiver タイミングモデル

Receiver は、Sender からのスナップショットとローカルクロックを組み合わせてタイミングを再構築します。

典型的な Receiver 処理パイプライン：

1. INTTC パケット受信
2. シーケンス番号検証
3. ACTIVE によるソース解決
4. タイミングアンカー確立
5. ローカルクロックによる時間進行

Receiver はパケットを **フレーム更新ではなくタイミングアンカー** として扱うべきです。

このモデルにより Receiver は以下を実現できます。

- パケットジッターの吸収
- 軽微なパケットロス耐性
- 連続した時間進行の維持

Receiver は必要に応じて次のような戦略を実装できます。

- パケットロス時の Holdover
- 再取得時のスムージング
- 外部リファレンスによるドリフト補正

具体的なアルゴリズムは実装依存であり、プロトコルでは厳密に定義しません。

## 7.3 パケットエンコードモデル

INTTC v1.2 は **可読性と実装容易性を重視した text ベースの key=value フォーマット**を採用しています。

パケット形式：

- UTF‑8 テキスト
- ASCII 安全文字推奨
- `|`（パイプ）区切り
- `KEY=VALUE` フィールド

一般的なパケット例：

```text
INTTC1|VER=1|SEQ=10452|SRC_ID=RSV001|STATE=PLAY|SRC_TC=01:23:45:12|ALT_TC=01:23:45:12|ACTIVE=SRC|DISPLAY=SRC|OUTPUT=SRC|FPS=23.976|DF=0
```

エンコードルール：

1. パケットは次の識別子で開始する必要があります。

```text
INTTC1
```

2. 各フィールドは次の形式です。

```text
KEY=VALUE
```

3. フィールド区切りは

```text
|
```

4. フィールド順序は推奨レイアウトに従うことが望ましいですが、Receiver は **キー名で解析**する必要があります。

5. 未知のフィールドは無視して構いません。

6. 任意フィールドが欠落していても、パケットは無効とはみなされません。

設計意図：

- ログで人間が読める
- パケットキャプチャでデバッグしやすい
- Python / Go / JavaScript / 組込みC で生成しやすい
- 後方互換性を保った拡張が可能

このテキストエンコードが INTTC v1.2 の **正式なパケット表現**です。

## 7.4 クロック権威モデル

INTTC は **Sender 権威型タイミングモデル（sender‑authoritative model）** を採用します。

このモデルでは：

- Sender がタイムラインの権威的ソースとなる
- Receiver はパケットスナップショットとローカルクロックで時間を追従する

Receiver はローカルシステムクロックを主基準としてパケットを再解釈してはいけません。

パケット内のタイムコードが **権威的タイミング状態**となります。

ローカルクロックは次の用途のみに使用されます。

- パケット間の時間進行
- パケット到着ジッターの吸収

この設計により、以下のような異種システム間でも一貫したタイムコード解釈が可能になります。

- NLE ソフトウェア
- ステージ表示システム
- LTC Bridge ハードウェア
- モニタリング Receiver

PTP などのクロック同期技術は **実装拡張**として使用できますが、INTTC コアプロトコルの定義外です。
