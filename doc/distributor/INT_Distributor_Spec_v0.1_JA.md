


# INT Distributor 仕様 v0.1

Status: Draft

---

# 1 概要

INT Distributor は INT Platform の中心となるルーティングコンポーネントです。

その役割は、1つまたは複数の Sender から送信された INTTC パケットを受信し、
それらを複数の Receiver に対して安定かつスケーラブルに配信することです。

Distributor は **ステートレス、もしくは最小限の状態のみを持つルーティングレイヤー**
として設計されています。これにより、クリエイティブソフトウェアは軽量なまま、
ネットワークインフラ側で配信と同期を処理できます。

Distributor は **タイムコードを生成しません**。
また **タイムライン状態を変更することもありません**。

Distributor の責務は以下に限定されます。

- パケット受信
- パケットルーティング
- 接続管理

---

# 2 システム内での役割

INT Platform アーキテクチャにおいて、Distributor は Sender と Receiver の
間に配置されます。

典型的なトポロジーは次の通りです。

```
Creative Software Timeline
        ↓
INT Sender
        ↓
INT Distributor
        ↓
Receivers / Bridges / Monitoring Systems
```

ネットワーク上には複数の Sender が存在する可能性があります。

また複数の Receiver が同一 Distributor に接続することも可能です。

Distributor を導入することで、Sender が各 Receiver と直接通信する
必要がなくなり、タイミングデータの配信を中央集約型にできます。

---

# 3 設計目標

Distributor は次の設計目標に基づいて設計されています。

- **複数 Sender / 複数 Receiver** をサポート
- INTTC パケットの **中央ルーティング** を可能にする
- Sender → Receiver 間の **低遅延** を維持
- 可能な限り **トランスポート非依存**
- タイミングデータの **変更を行わない**
- 大規模モニタリング環境への対応

重要な原則として、Distributor は

**タイミングの権威（Timing Authority）になってはいけません。**

タイミングの権威は常に Sender 側にあります。

---

# 4 Distributor の責務

Distributor は次の処理を行います。

1. Sender からパケットを受信
2. Sender の存在情報を軽量に記録
3. パケットを Receiver に転送
4. Receiver 接続を管理
5. 必要に応じて監視インターフェースを提供

Distributor は次の処理を行ってはいけません。

- タイムコード値の書き換え
- タイミング状態の再解釈
- タイミング補間
- 同期の権威として振る舞うこと

---

# 5 Sender 登録

Sender は UDP を使用して INTTC パケットを送信します。

Distributor がパケットを受信すると、Sender に関する
最小限のメタデータを記録します。

例：

- sender_id
- 最終パケット受信時刻
- 送信元アドレス

これらの情報は監視やデバッグ目的で利用される場合がありますが、
パケットルーティングの動作には影響してはいけません。

Sender の存在は **パケットベース**で判断されます。

永続的な登録プロトコルは必要ありません。

---

# 6 Receiver サブスクリプション

Receiver は次の方法のいずれかで Distributor に接続します。

- WebSocket
- TCP ストリーム
- UDP リスナー

実装により接続方式は異なる場合がありますが、論理動作は同じです。

Receiver は INTTC パケットのストリームを購読します。

Distributor は受信したパケットをすべての
アクティブな Receiver に転送します。

Receiver は自身でタイミングを再構築する責任を持ちます。

---

# 7 パケットルーティング

Distributor が INTTC パケットを受信した場合、
次の処理を行います。

1. ネットワークからパケット受信
2. 基本的な構造を検証
3. Sender メタデータを記録
4. アクティブ Receiver へ転送

Distributor は **パケット内容を変更してはいけません。**

転送は可能な限り迅速に行う必要があります。

推奨動作：

- ノンブロッキング転送
- 非同期配信

---

# 8 複数 Sender の扱い

ネットワーク上では複数の Sender が同時に動作する可能性があります。

Receiver はどの Sender を使用するかを
自分で選択できます。

そのため Distributor は

**複数 Sender のパケットが同時に存在する状態**

を許容する必要があります。

Distributor は Sender 選択ロジックを
強制してはいけません。

どの Sender を権威とするかは
Receiver ソフトウェアが決定します。

---

# 9 障害時の動作

Sender がパケット送信を停止した場合、
Distributor は特別な復旧処理を行いません。

Sender の消失は
Receiver 側のタイムアウトロジックにより
検出されます。

Distributor は単に
その Sender のパケットが届かなくなるだけです。

この設計により、ルーティングレイヤーの
複雑化を防ぎます。

---

# 10 レイテンシモデル

Distributor は **最小限の追加レイテンシ**
しか導入しないことが望まれます。

典型的なパイプライン：

```
Sender → Distributor → Receiver
```

可能な限りバッファリングを行わず、
即時転送を行う必要があります。

推奨レイテンシ寄与：

```
< 1 ms
```

実際の遅延はネットワークインフラに依存します。

---

# 11 監視インターフェース

Distributor は任意で次のような監視機能を
提供する場合があります。

- HTTP ステータス API
- Web ダッシュボード
- パケット統計

例：

- アクティブ Sender 数
- アクティブ Receiver 数
- パケットレート
- 最終パケット受信時刻

これらの監視インターフェースは
読み取り専用である必要があります。

パケットルーティングの動作に
影響してはいけません。

---

# 12 セキュリティ考慮

実運用環境では Distributor に
次のようなセキュリティ機構を
導入する場合があります。

- アクセス制御
- ネットワーク分離
- 認証付き Receiver 接続

ただし、これらの機構は
INTTC パケット内容を変更してはいけません。

---

# 13 INTTC プロトコルとの関係

Distributor は **トランスポートインフラ**です。

INTTC プロトコル自体を再定義するものではありません。

INTTC パケット構造およびタイミングセマンティクスは
次の仕様で定義されています。

```
INTTC Protocol Specification
```

Distributor はこのプロトコルに準拠した
パケットをルーティングするだけです。
