# INT Platform
## 設計思想 (Philosophy)

INT Platform は、現代の編集ソフトウェアに不足していた機能を補うための実用的なツールとして始まりました。

映像制作の現場では **タイムコードによる同期** が重要な役割を果たしていますが、多くの現代的な制作ツールはソフトウェア内部で完結する設計となっており、タイムラインのタイムコードを外部システムへ簡単に配信する仕組みを持たないことが多くあります。

INT Platform は、この問題を解決するために **ネットワークネイティブなタイムコード配信レイヤー** を導入し、ソフトウェアタイムライン、モニタリングツール、そして従来の LTC ベースのハードウェア機器を共通のタイミングシステムで動作させることを目的としています。

# Origins

INT Platform は、実際の編集ワークフローの中で生まれました。

プロジェクトはまず、DaVinci Resolve の編集作業において不足していた機能を補うツールの開発から始まりました。これらのツールは、日常のポストプロダクション作業の中で発生する実務的な問題を解決するために作られたものです。

ツールの発展とともに、次のような要素が必要になりました。

- 編集タイムラインのタイムコードを外部へ送信する仕組み
- ネットワークタイムコードを LTC に変換する Bridge デバイス
- タイムコードを監視する Receiver アプリケーション

この流れの中で、自然にネットワークベースのタイムコードシステムの設計へと発展していきました。

つまり INT Platform は、最初からプロトコル設計を目的として始まったプロジェクトではありません。実際の制作ツールとワークフローから自然に生まれたシステムです。

---

# 1. ソフトウェアとハードウェアの橋渡し

現代の制作ツールの多くはソフトウェア環境で動作します。

例：

- NLE（DaVinci Resolve など）
- Unreal Engine
- メディアサーバー
- バーチャルプロダクションシステム

これらのシステムはネットワーク環境との親和性が高い一方、従来の同期インフラとは直接接続できないことが多くあります。

一方で、放送や舞台の現場では今でも **LTC（Linear Timecode）ベースの同期システム** が広く使われています。

例：

- 放送設備
- 舞台照明システム（DMX）
- メディア再生ハードウェア
- タイムコードディスプレイ
- 各種プロフェッショナル機器

INT Platform は、ネットワーク経由でタイムコードを配信し、必要に応じて **LTC 信号へ変換する Bridge デバイス** を介して、これらのシステムを接続します。

```
Software Systems
       ↓
      INTTC
       ↓
     Bridge
       ↓
       LTC
       ↓
Legacy Hardware
```

---

# 2. ネットワークによるタイムコード配信

INT Platform は既存の同期システムを置き換えることを目的としていません。

代わりに、既存のインフラの上に **ネットワークベースのタイムコード配信レイヤー** を追加します。

これにより次のような構成が可能になります。

- ソフトウェアタイムラインからタイムコード生成
- ネットワーク経由で複数デバイスへ配信
- Bridge デバイスによる LTC 変換
- 従来機器との互換性維持

この構造により、既存ワークフローを壊すことなく柔軟なシステム設計が可能になります。

---

# 3. 異なる同期要求への対応

制作環境によって、必要とされる同期精度は大きく異なります。

編集環境では次の要素が重視されます。

- 応答性
- 柔軟性
- 実装のシンプルさ

一方、次のような環境では **フレーム単位で安定した同期** が必要になります。

- バーチャルプロダクション
- LEDウォール
- Unreal Engine
- 舞台照明システム
- メディアサーバー

この違いに対応するため、INT Platform では **2種類の Output Policy** を定義しています。

```
Loose
Tight
```

### Loose

編集・モニタリング用途向け。

特徴：

- パケット駆動更新
- 小さなタイミング揺らぎを許容
- 再同期を高速に行う

用途例：

- クライアントモニター
- 編集レビュー
- ソフトウェア表示

### Tight

同期精度が重要な環境向け。

特徴：

- 連続したフレーム進行
- 安定したフレーム境界
- クロック制御された出力

用途例：

- Unreal Engine 同期
- LEDウォール再生
- 舞台照明システム
- メディアサーバー

---

# 4. ソフトウェア定義タイミング

INT Platform はタイムコード処理を **ソフトウェア定義システム** として扱います。

Receiver は内部的に時間を次の3層に分離して処理します。

```
Packet Time
Resolved Time
Output Time
```

この構造により、以下が可能になります。

- ネットワークジッター耐性
- 出力の連続性維持
- パケット途絶時の安定動作
- 異なる同期環境への適応

---

# 5. 互換性を最優先する設計

INT Platform の重要な設計原則は次の通りです。

> 既存ワークフローとの互換性を犠牲にしない

そのため、**LTC 出力は常に第一級のインターフェース** として扱われます。

LTC は現在もなお、

- 放送
- 舞台
- メディア再生
- 映像制作

など、多くの現場で使われている最も普遍的な同期信号です。

INT Platform はこのエコシステムと共存することを前提に設計されています。

---

# 6. 将来の同期システムへの基盤

INT Platform は特定の実装に限定されるものではありません。

将来的には次のような拡張を想定しています。

- PTP（Precision Time Protocol）同期
- GPS同期
- マルチ Sender 環境
- 分散同期システム

プロトコル伝送、時間解決、出力生成を分離することで、INT Platform は将来のタイミング技術の基盤として機能する設計になっています。

---

# 7. オープンプラットフォーム思想

INT Platform はオープンプラットフォームの思想に基づいて設計されています。

このプロジェクトの目的は、ネットワークベースのタイムコード配信に関する **概念・アーキテクチャ・プロトコル** を共有することで、業界全体が共通の理解を持てる基盤を作ることです。

そのため、以下の要素は公開されることを前提としています。

- プロトコル仕様
- システムアーキテクチャ
- タイミングモデル
- 設計思想

一方で、INT に対応した具体的なツールの実装は、必ずしも公開される必要はありません。

タイムラインツール、Bridge デバイス、Receiver アプリケーションなどの製品は、INT の概念を利用しながら、それぞれ独自の内部設計や最適化を持つことができます。

INT Platform は次の二つの役割を持つことを目指しています。

- ネットワークタイムコードシステムのためのオープンな技術基盤
- その基盤の上で構築されるプロフェッショナルツールのプラットフォーム

つまり INT は、**システムの考え方と言語を共有しつつ、具体的なツールの実装には自由度を残す**ことを目指しています。

# 8. 設計原則

INT Platform の開発は、いくつかの実践的な設計原則に基づいています。

- **実際のワークフローを最優先する** — 抽象的なシステム設計よりも、まず現場の問題を解決するツールであること。
- **再発明より互換性を重視する** — LTC など既存インフラを活かす設計を優先する。
- **ネットワークネイティブな発想** — 現代の IP ベースの制作環境に自然に統合できるタイミングシステムを目指す。
- **概念は共有し、実装は独立する** — アーキテクチャとプロトコルは公開しつつ、各ツールの実装は独立して進化できるようにする。

これらの原則は、INT Platform が実際の制作現場から生まれたプロジェクトであることを反映しており、今後の発展の指針となります。
